01The core idea

The lead agent runs the org. It doesn't do all the work.

Work flows down from you and reports back up. Discovery fans out to specialists; consequential decisions funnel back to one accountable human.

CorrectnessSecurityMaintainabilityUX
Human approval & releasethe decision Codex never makes alone
What the lead agent actually does
  1. Understand the objective
  2. Inspect the repository and documentation
  3. Create an implementation plan
  4. Break work into isolated, testable tasks
  5. Delegate parallelizable research and development
  6. Integrate the outputs
  7. Run verification — with evidence
  8. Request independent reviews
  9. Report decisions, risks and remaining work

Autonomy inside boundaries beats full autonomy.

High agent independence within clearly defined scope, with humans owning the consequential calls. That's the whole model — everything below is how to hold it.

Using the top tier for everything

Don't spend Sol on renaming variables, straightforward CRUD endpoints, standard unit tests, or reformatting config. It costs more without making simple work more correct.

Spend the reasoning where the mistakes are expensive.

Escalate to Sol when getting it wrong causes rework, data loss, or a security hole. Everywhere else, let Terra and Luna carry the volume.

Step 0 · Define

Define the outcome before touching code

The first task is never "build the app." It's a Project Definition Package: who the user is, the problem, what success means, and what is explicitly not being built.

First taskAct as lead product engineer. Don't write code. Inspect all project files, create a Project Definition Package under docs/product/, separate facts from assumptions and open questions, define measurable success, and state what's out of the first release. Mark uncertain info as an assumption.
Step 1 · Research

Research before architecture

Four streams — market, users, technical, and the existing systems (the old CRM, spreadsheets, WhatsApp, the door system). Every finding ends in a decision: build, buy, integrate, defer, reject, or investigate.

Research briefAct as the product & competitive research agent. Identify table-stakes vs differentiating capabilities for a modern gym platform, and where customers abandon booking. Cite every external claim with a confidence level. Do not change code. End with the five biggest implications and a build/buy/integrate/defer call for each finding.
Step 2 · Requirements

Turn research into a PRD

Strong user stories carry acceptance criteria. Weak: "users can book courses." Strong: a parent sees only age-appropriate courses for each child, conflicts surface before confirmation, and a failed notification never reverses a successful booking.

PRD promptAct as senior PM. From the approved research, produce a PRD for the first releasable version. Per feature: target user, problem, business outcome, acceptance criteria, analytics events, dependencies, privacy/security implications, and a Must/Should/Could/Not-Now class. Don't include a feature just because a competitor has it. No code.
Step 3 · Architecture

Design before implementation

One app or split customer/CRM front-ends? How are parents, children and memberships linked? How is double-booking prevented? Each consequential answer becomes an with a threat model.

Architecture promptAct as principal architect. Read docs/product, docs/research and the repo. Propose the simplest architecture that meets the requirements — avoid microservices unless scaling, ownership or isolation demands them. Define system boundaries, sources of truth, and a failure mode for every external integration. Produce a threat model and ADRs. Request review before implementation.
Step 4 · Repo

Prepare the repo for agentic work

Codex shouldn't rediscover your rules every session. A root AGENTS.md maps the repo; narrower files sit beside the code; repeatable jobs become skills. (Full detail in the next chapter.)

Step 5 · Milestones

Break the release into atomic tasks

Never "build the booking platform." A good task has one outcome, one bounded area, testable criteria, and is reviewable as a single diff. Foundation → discovery → booking → accounts → CRM → AI → hardening.

Better taskAdd a transactional booking service that prevents two confirmed reservations from consuming the same final course slot — with tests for the concurrent case. Scope to the booking domain only.
Step 6 · Implement

Implement with a controlled loop

Read → inspect → restate → plan → smallest coherent change → scoped tests → inspect the diff → broaden verification → update docs → report. Checkpoints for risky work stop an agent building the wrong thing for an hour.

Step 7 · Test

Testing is designed, not appended

Unit for rules and permissions, integration for transactions and adapters, end-to-end for the journeys that matter — register, book, cancel, convert a lead. An independent test agent tries to disprove "done."

Step 8 · Review

Review before you integrate

One "review my code" pass is too broad. Run focused lenses — correctness, security, maintainability, performance, UX — then integrate the milestone against a clean database and a real migration. (Detail in chapter 07.)

Step 9 · Release & monitor

Ship on purpose — then watch

Local → CI → preview → staging → limited rollout → full. Gates: tests, review, security, migrations verified, rollback possible, owner approves. After launch, monitor the signals that matter and let automations do triage — never the deploy.

Weekly automationReview the last 7 days: merged PRs, repeated review feedback, flaky tests, doc drift, new TODOs, recurring mistakes, dependency alerts. Recommend AGENTS.md improvements, new skills and missing checks. Do not change production code. Produce a concise engineering-system improvement report.

04The biggest efficiency step

Give the org a memory it doesn't have to rebuild.

A root should tell Codex how to navigate, how to run tests, and which conventions hold — as a map, not a 30-page manual. Put narrower rules in nested files close to the code they govern.

AGENTS.md says what is stable.

Project purpose, repository map, required workflow, commands, and a crisp definition of done that every session inherits.

A skill says how to do a repeatable job.

Adding a database-backed feature, creating an endpoint, running a release. Keep each skill focused, with explicit inputs and outputs — and only package a workflow after you've done it successfully.

⌁Use ChatGPT Work for broad multi-source investigation, Codex when the task touches the repository, MCP for direct access to outside docs and tools, and plugins to package skills, connectors and MCP tools into a reusable workflow.
05The unit of real work

A role isn't a sentence. It's a contract.

"Be a senior backend developer and implement waiting lists" is a wish. A task contract is a specification an agent can be held to — and reviewed against.

Agent task contract
ROLE Backend engineer for the booking domain. OBJECTIVE Implement cancellation and waiting-list promotion for course bookings. CONTEXT Read AGENTS.md, docs/product/booking.md, ADR-007-booking-concurrency.md, and the existing booking service and tests. SCOPE — you may change services/api/src/booking/ services/api/tests/booking/ packages/contracts/src/booking/ Do NOT change auth, unrelated tables, frontend styling, or payment behavior. REQUIREMENTS 1. A customer may cancel an eligible confirmed booking. 2. Cancellation releases capacity transactionally. 3. The oldest eligible waiting-list entry is promoted. 4. Promotion is idempotent. 5. Notification failure must not roll back booking state. 6. All state changes produce audit events. EDGE CASES simultaneous cancellations · already-cancelled booking · customer no longer eligible · notification provider down · cutoff crossed · invalid waitlist entries. VERIFICATION Add unit + integration tests. Run lint, typecheck and the relevant suites. Do not claim success unless commands complete. OUTPUT Summary · files changed · decisions · commands run + results · unresolved risks · suggested follow-up.

Require evidence

"Everything works" is not a result

Make the agent report the commands it ran and what passed — and, just as important, what it did not verify (production notification provider, load above 200 concurrent cancels).

The controlled implementation loop

Read instructions → inspect code → restate the task → name assumptions → small plan → smallest coherent change → scoped tests → inspect the diff → fix → broaden verification → update docs → completion report. Add checkpoints (plan → contracts & tests → implementation → security → integration) for anything high-risk.

06Prove it, don't append it

The pyramid is a plan, not an afterthought.

Design the layers up front, then hand the finished feature to an agent whose only job is to prove it isn't done.

End-to-endregister · book · cancel · convert a lead
Integrationdb · transactions · adapters · API contracts
Unitrules · validation · permissions · state transitions
Non-functionalload · accessibility · security · failure injection · backup restore · migration tests
codex · adversarial test
Try to break it…
Independent test agent
Act as an adversarial test engineer. Do not modify the implementation yet. Read the requirements and try to disprove that the feature is complete: missing acceptance criteria, edge cases, concurrency, invalid states, permissions, error handling, data integrity, accessibility, observability. Produce findings first. Only add tests after documenting the gaps.
⌁The agent that wrote the feature is biased toward its own reading of the spec. A separate agent, prompted to refute, catches what self-review never will.
07Many lenses, one report

Split "review my code" into five sharp questions.

Each reviewer gets one lens and one job. The coordinator deduplicates, verifies evidence, and ranks by impact — not by volume.

1
Correctness

Diff vs the stated acceptance criteria — missing or partial behavior

2
Security

Auth, location/tenant boundaries, input validation, secrets, injection

3
Maintainability

Complexity, duplication, hidden coupling, weak names, module boundaries

4
Performance

Query behavior, indexes, repeated calls, caching, concurrency

5
UX & accessibility

Loading / empty / error / success states, keyboard, focus, contrast

🛡️Codex Security can scan the repo or a specific change, validate the vulnerability, and propose a patch — run the standard scan before the deeper one. Apply findings, then re-review to confirm they're resolved.
Never

Let the implementer approve itself

The author's review is the weakest review. Independent correctness, security, maintainability and UX passes are what make a diff trustworthy.

08Access & ownership

Give each agent only the tools its job needs.

A documentation agent doesn't need production database access. A marketing agent doesn't need merge rights. A test agent rarely needs deploy credentials. More permissions do not mean more effectiveness.

Human-owned decisions

  • Business objective & budget
  • Product priorities & major trade-offs
  • Acceptable risk & personal-data policy
  • Production credentials
  • Final UX & security acceptance
  • Release authorization

Agent-owned execution

  • Repository exploration & plans
  • Routine coding & test generation
  • Migrations & documentation
  • Structured research
  • Repetitive refactoring & code review
  • Incident investigation & release prep
🔑For anything near production, insist on least privilege: explicit approvals, sandboxing, restricted credentials, short-lived tokens, protected branches, and audited pipelines. A sensible Codex default is approval-on-request, not unrestricted command execution — and cloud routines run without approval dialogs, so strip every permission the job doesn't need.
09The twelve efficiency killers

The failures that quietly waste the most.

Almost every stalled Codex project traces back to one of these. Fix the system, not just the next agent.

01

Starting with code

Research, requirements and architecture stay implicit — so the agent invents hundreds of small assumptions.

02

One enormous task

Huge prompts make huge diffs that are hard to test, review and reverse.

03

Top model for everything

Burns capacity without making straightforward work any more correct.

04

Every agent the same role

Five "senior engineers" aren't a team. Give each a distinct responsibility and output.

05

Parallelizing overlapping edits

Parallel work is only faster when the boundaries are actually stable.

06

No persistent instructions

Without AGENTS.md, conventions get re-explained every session and drift.

07

No reliable test command

An agent can't prove its work when there's no deterministic verification path.

08

Self-approval

Letting the implementation agent sign off on its own work. Use independent reviewers.

09

Activity mistaken for progress

Ten agents can produce ten documents without resolving one critical decision.

10

Undocumented assumptions

Every significant assumption should be visible in a decision or assumption log.

11

Unrestricted tool access

Broad permissions add risk, not capability.

12

Never updating the system

When agents repeat a mistake, fix AGENTS.md, the skill, the lint rule, the test — not just the next prompt.

10Put it all together

The prompt that starts a serious milestone.

One lead agent, coordinating — not implementing everything itself. This is the repeatable backbone of the whole playbook.

Master orchestration prompt
You are the lead engineering agent for this repository. Coordinate the work — do not immediately implement everything yourself. First: 1. Read all applicable AGENTS.md files. 2. Read the product requirements, architecture decisions and current milestone. 3. Inspect the relevant structure and existing tests. 4. Identify facts, assumptions, blockers and unresolved decisions. 5. Produce a dependency-aware implementation plan. Then: 6. Divide the work into atomic tasks. 7. Identify which can safely run in parallel. 8. Assign specialist subagents only where responsibilities don't overlap. 9. Give each an explicit scope, allowed files, required outputs and verification. 10. Keep shared contracts and architecture decisions under your control. During execution: 11. Require agents to run relevant checks. 12. Accept no claim without command results or inspectable evidence. 13. Integrate in dependency order. 14. Resolve contract inconsistencies before continuing. 15. Maintain a decision log and risk register. Before completion: 16. Run the full milestone verification. 17. Request independent correctness, security, maintainability and UX reviews. 18. Resolve blocking findings. 19. Update documentation. 20. Produce a final report: done · not done · files changed · checks run · decisions · accepted risks · unresolved issues · next milestone. Do not hide uncertainty. Do not silently expand scope. Do not deploy to production without explicit human authorization.
