04Reusable judgment

Turn your release habits into a skill.

A project skill lives in .claude/skills/name/SKILL.md. Claude can load it when relevant, or you can invoke it directly with /name.

Create the skill
Create a focused project skill named gather-release. It should gather the current change summary, run the documented tests, verify migrations and environment-variable names, check the guest flow in a narrow viewport, and produce a release recommendation. It must never deploy or change data. Keep SKILL.md concise and put longer checklists in a referenced file.

Do not turn every prompt into a skill.

Package a workflow only after you have performed it successfully and expect to repeat it.

06Optional outside context

Bring in calendars without giving away the steering wheel.

A connector is an MCP server with a friendly setup flow. For Gather, the useful first move is read-only: compare candidate dates with your own calendar, then draft—not send—an invitation.

Connected planning
Using my connected calendar, check only my availability for Gather’s three candidate dates. Do not inspect unrelated event details, create events, invite anyone, or send messages. Return busy/free windows and draft a short invitation I can review.

07Rules that actually run

Add one narrow guardrail.

Hooks fire at specific Claude Code lifecycle events. Unlike a reminder in CLAUDE.md, a command hook executes automatically—so start small, keep it fast, and understand every command.

PreToolUseCheck or block before a matching tool runs.
PostToolUseValidate after a successful tool action.
Failure eventsReport or guide recovery after a tool fails.
Guardrail prompt
Propose one project hook for Gather: after Claude edits application files, run the existing fast formatter and focused test command. Show the exact configuration and commands before writing them. It must not modify the database, deploy, install packages, or hide failures. Add a short disable-and-debug note.
Hook failure

The guardrail blocks every edit

Disable the hook, run its command directly, fix the matcher or command, then re-enable it. Never weaken all permissions just to silence one broken hook.

08Useful while unattended

Automate the preparation, not the decision.

A safe Gather routine checks who has not replied and prepares a reminder draft. It does not message guests, alter RSVPs, or push unrestricted changes.

Routine design
Design a weekly Gather follow-up routine that reads only the event’s RSVP summary, produces a private report, and drafts reminder text for my review. It must not send messages, change event data, or push to unrestricted branches. Include the minimum repository, network, variable, and connector access it needs. Do not create it until I approve the design.
⚠Cloud routines run without approval dialogs. Remove every repository, connector, network permission, and write capability the job does not require.
09Many lenses, one report

Run the quality audit without creating a panic list.

Give each reviewer a narrow lens. Then ask the coordinator to deduplicate findings, verify evidence, and rank impact—not volume.

A11Y
Accessibility

Keyboard, focus, labels, contrast, reduced motion

SEC
Privacy and security

Secrets, input handling, destructive paths, exposure

PERF
Performance

Measured slow paths and unnecessary payloads

UX
Responsive usability

Guest tasks across narrow and wide screens

Coordinated audit
Coordinate four read-only reviewers for accessibility, privacy/security, measured performance, and responsive usability. Each finding needs evidence, affected user, severity rationale, and a verification step. Merge duplicates and challenge unsupported claims. Return at most eight verified findings; do not edit code.

“Eight verified findings” beats “forty-seven concerns.”

A bounded report forces the crew to distinguish impact from possibility.

10The release room

Bring every result back to one decision.

The coordinator owns the release narrative: what changed, what was proven, what remains uncertain, how to deploy, and how to go back.

Release prompt
Run /gather-release for the current candidate. Reconcile the specialist reports, inspect the final diff, run the full test suite and browser smoke path, verify the production migration and rollback point, then present a go/no-go recommendation. Do not deploy until I explicitly say “go.”
No-go is success

A migration test fails five minutes before release

Do not “try the deploy and see.” Keep production unchanged, capture the failure, narrow the fix, rerun the complete gate, and make a fresh decision.

11After launch

Watch one signal and improve one thing.

Do not turn a small event into an analytics empire. Logs, one organizer observation, and one guest comment are enough for the first improvement.
