Codex · practical reference

Commands matter. Context matters more.

Use this page beside a real repository. Each move tells Codex what to do, when to use it, and what evidence should come back before you accept the work.

Codex referenceVerified 18 Jul 2026Official manual
Start · choose the work mode

Open Codex on the repository.

Start in the project directory, then give one bounded goal with context, constraints, and a clear finish line.

CLI
codex -m gpt-5.6
Use this when: beginning an interactive local task.Watch for: the wrong working directory or missing project instructions.
Plan · inspect before editing

Make ambiguity cheap.

Plan mode lets Codex gather context, ask questions, and propose the work before it changes files.

Interactive command
/plan
Use this when: the task crosses systems, has unclear requirements, or could be risky.Shortcut: Shift+Tab also toggles Plan mode in supported clients.
Memory · durable repository guidance

Give the project an operating map.

AGENTS.md loads into context automatically. Keep it short: layout, commands, conventions, constraints, and what “done” means.

CLI scaffold
/init
Use this when: a repository needs repeatable guidance across tasks.Watch for: vague essays. Narrower AGENTS.md files should live closer to the code they govern.
Delegate · parallel agents

Split independent work, not one tangled edit.

Ask directly for subagents when exploration, tests, security review, and maintainability review can run independently.

Delegation prompt
Review this branch with parallel subagents. Use one for correctness, one for security, and one for missing tests. Wait for all three, then return one prioritized report with file references.
Inspect threads: use /agent in the CLI.Watch for: parallel write-heavy tasks that compete for the same files.
Reuse · skills, plugins, connectors

Package work that repeats.

Use a skill for a reusable workflow, a plugin for a shareable bundle, and an MCP server or connector for live outside data and actions.

Design prompt
We repeat this release audit every week. Propose the smallest reusable Codex surface for it: prompt, AGENTS.md rule, skill, plugin, connector, hook, or automation. Explain the boundary before creating anything.
Use this when: a successful pattern should stop living in memory.Watch for: turning a one-off instruction into permanent configuration.
Safety · permissions and sandbox

Choose authority before delegation.

Approval mode controls when Codex asks; sandbox mode controls what the command can read or write. Subagents inherit the parent turn’s live policy.

Interactive command
/permissions
Use this when: the task needs a different permission boundary.Default: keep access tight and expand it only for a concrete need in a trusted repository.
Review · inspect the change

Ask for a second pass before acceptance.

Review uncommitted changes, a commit, or the branch against its base. Then inspect the diff yourself.

Interactive command
/review
Use this when: implementation is complete but not yet trusted.Require: prioritized findings, file references, and explicit test gaps.
Speed · faster supported models

Spend more credits only when latency matters.

Fast mode increases supported model speed by 1.5× and consumes credits at a higher rate. It is different from choosing the separate Spark model.

CLI controls
/fast on /fast status /fast off
Use this when: short iteration time is worth the higher credit multiplier.Watch for: leaving fast mode on for long, low-urgency runs.
Verify · define done

Return evidence, not reassurance.

Codex works best when the prompt names the checks and the final response reports what actually ran.

Verification prompt
Implement the approved change. Before finishing, run the relevant tests and lint, inspect the final diff, test the user-visible behavior, and report each command and result. If a check cannot run, say exactly why.
Use this when: every implementation task.Watch for: “should work” without executed checks.
Recover · preserve user work

Diagnose before undoing.

When the direction is wrong, inspect the working tree and separate your changes from existing user work before reverting anything.

Recovery prompt
Stop editing. Show git status and the relevant diff, identify which changes came from this task, and propose the smallest recoverable rollback. Preserve all unrelated user changes. Do not use a hard reset.
Use this when: a change drifted or a test exposed the wrong approach.Watch for: destructive resets and broad file replacement.
Ship · keep the human gate

Prepare delivery; do not hide the decision.

Have Codex summarize scope, verification, risks, and the exact external action before committing, pushing, deploying, or opening a pull request.

Delivery prompt
Prepare this change for delivery. Summarize the diff, checks, remaining risks, and proposed commit message. Stop before any push, deployment, or pull-request action unless I explicitly asked for it.
Use this when: the code is ready to leave the local workspace.Watch for: treating a passing build as automatic release approval.
Automate · non-interactive run

Use a bounded command for CI-shaped work.

Codex exec is useful for scripted review and automation when the prompt, permissions, and expected output are already stable.

CLI
codex exec -m gpt-5.6 "Review the current changes and report correctness risks with file references."
Use this when: the task is repeatable and does not need live clarification.Watch for: jobs that require an approval dialog; non-interactive runs cannot invent new authority.
